## 1. HairTexture（头发纹理检测）

**问题定义**：头发区域在生成/编码过程中高频纹理信息丢失，表现为发丝边界模糊、纹理退化为块状色块、出现棋盘格或波纹伪影。本质是空间高频分量被压缩或生成模型未能恢复，导致头发区域从"可辨识单根发丝"退化为"低频色团"。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| FFT 高频能量分析 | 对头发 ROI 做 2D FFT，统计高频段（>阈值频率）能量占总能量比值；高频占比低 → 纹理丢失 | ✅ 选用 | 直接量化纹理丰富度，与 MOS 标注中"发丝清晰可辨"高度对齐；输出高频能量比可直接解释为"纹理保留率"，无需训练数据；计算开销 <5ms/帧 |
| BRISQUE / NIQE 无参考质量评估 | 基于自然场景统计（NSS），提取 MSCN 系数的统计特征，与预训练模型做距离度量 | ❌ 淘汰 | 面向整图通用质量，对"头发纹理"这一局部特征缺乏针对性；NIQE 在合成图像上 NSS 先验不成立，分数与 MOS 标注相关性 <0.3；无法输出"哪种纹理退化"的可解释 evidence |
| CNN 二分类器（好/差头发） | 收集头发区域 patch，标注好/差，训练 ResNet-18 等轻量分类器 | ⏳ 备选 | 精度天花板高，但需要大量头发区域标注数据（当前无此数据集）；黑箱输出一个概率值，可解释性不足；后续数据积累后可作为 FFT 方案的增强校验 |
| Gabor 滤波器组 | 多方向多尺度 Gabor 滤波，统计各方向响应能量，判断纹理是否存在方向性结构 | ❌ 淘汰 | 对发丝走向敏感，但头发方向多变导致需要大量滤波器（典型 6 方向 × 4 尺度 = 24 个），计算量高；在发丝弯曲/交叉区域误判率高；超参（方向数、尺度）需按发型调整，鲁棒性差 |

**选型依据**：MOS 标注标准中，发丝清晰可辨 ≥ 4.0、糊成一片 ≤ 3.8，核心区分点在于高频纹理的有无。FFT 高频能量比与此标准天然对齐——能量比高意味着发丝纹理保留完好，能量比低意味着退化为色块。该方案无需训练数据，可解释性强（直接输出能量比数值和频谱热力图），且计算效率满足逐帧检测需求。

**输出示例**：
```json
{
  "detector": "HairTexture",
  "score": 3.7,
  "confidence": 0.85,
  "evidence": {
    "method": "fft_high_freq_energy",
    "roi": "hair_region",
    "high_freq_ratio": 0.12,
    "threshold": 0.25,
    "interpretation": "高频能量比 0.12 远低于阈值 0.25，头发区域纹理严重丢失，发丝不可辨识",
    "spatial_detail": {
      "left_hair": {"high_freq_ratio": 0.14, "status": "degraded"},
      "right_hair": {"high_freq_ratio": 0.10, "status": "severely_degraded"}
    }
  }
}
```

## 2. FaceArtifact（面部劣化检测）

**问题定义**：面部在生成过程中出现结构性失真或视觉异常，表现包括：表情僵硬/不自然（恐怖谷效应）、面部曝光过度发白、五官比例失调、皮肤纹理异常。本质是面部特征的高维语义表示偏离了真实人脸的分布空间。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| ArcFace embedding 偏移 | 用预训练 ArcFace 提取面部 embedding，计算与参考帧/人脸质量中心的余弦距离；距离大 → 面部异常 | ✅ 选用 | ArcFace 在训练过程中学到了人脸的高维语义结构，对恐怖谷、五官变形等语义级劣化天然敏感；embedding 偏移量可直接量化"面部偏离正常程度"；预训练模型现成可用（InsightFace），无需额外标注；偏移向量可做 PCA 分析定位劣化维度 |
| 关键点几何约束 | 用 68/98 点 landmark 检测器提取面部关键点，检查五官比例（眼距/脸宽、鼻长/脸高等）是否在合理范围 | ❌ 淘汰 | 仅能检测几何变形，对曝光发白、皮肤纹理异常等非几何劣化无感知；landmark 检测器在严重劣化帧上定位精度下降，导致级联错误；需手工定义大量几何比例阈值，维护成本高 |
| CLIP 图文相似度 | 用 CLIP 计算面部区域与正/负文本描述（"natural face" vs "distorted face"）的相似度差 | ⏳ 备选 | CLIP 对语义级异常有感知能力，可作为 ArcFace 的互补信号；但 CLIP 对微妙的面部劣化（如轻微曝光偏白）分辨力不足；推理开销大（ViT-L 约 30ms/帧）；可作为二级确认手段 |
| 训练劣化分类器 | 收集面部劣化样本（曝光、恐怖谷、五官异常），训练多标签分类器 | ❌ 淘汰 | 需要大量细粒度标注数据（当前仅有 MOS 整体分，缺乏面部劣化类型标注）；劣化类型持续变化（新模型新缺陷），分类器需频繁重训；输出类别概率的可解释性不如 embedding 偏移分析 |

**选型依据**：MOS 标注中面部曝光发白/恐怖谷扣 0.2~0.4，属于语义级劣化。ArcFace embedding 编码了人脸的身份和质量信息，偏移量天然反映面部"不自然程度"。与参考帧比较时，可区分"身份保持但质量下降"（小偏移+方向特定）和"面部严重失真"（大偏移），与 MOS 扣分梯度对齐。

**输出示例**：
```json
{
  "detector": "FaceArtifact",
  "score": 3.6,
  "confidence": 0.90,
  "evidence": {
    "method": "arcface_embedding_drift",
    "cosine_distance": 0.35,
    "threshold": 0.20,
    "drift_direction": "exposure_overblown",
    "interpretation": "面部 embedding 偏移 0.35 超过阈值 0.20，主要偏移方向对应曝光过度特征",
    "sub_scores": {
      "expression_naturalness": 0.82,
      "exposure_normality": 0.41,
      "feature_proportion": 0.88
    },
    "reference_frame": "frame_0001"
  }
}
```

## 3. EdgeBleed（边缘溢色检测）

**问题定义**：人物与背景的融合边界出现色彩异常，表现为绿幕残留（绿边）、Alpha 通道截断导致的黑边、抠图不精确导致的锯齿/毛刺、前景色与背景色在边界带的不自然混合。本质是 matting/compositing 阶段 Alpha 边缘处理不当，导致前景/背景色在过渡带产生色差。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| Lab 色差分析 + Alpha 边缘检测 | 在人体轮廓边缘带（由分割 mask 膨胀-腐蚀获取）转 Lab 色空间，计算 a/b 通道与邻域内侧的色差；绿边在 a 通道表现为显著负偏移 | ✅ 选用 | Lab 色空间解耦亮度与色度，色差计算直接对应人眼对溢色的感知；可精确定位溢色位置和颜色类型（绿/青/黑）；阈值物理含义明确（ΔE > 10 对应可察觉色差）；计算轻量且可逐像素解释 |
| RGB 通道统计 | 在边缘带统计 R/G/B 通道均值和比例，绿边 → G 通道占比异常高 | ❌ 淘汰 | RGB 空间亮度与色度耦合，绿色衣服/植物背景导致大量误报；无法区分"自然的绿色"和"溢出的绿边"；对黑边检测需额外逻辑（低亮度不等于黑边），规则碎片化严重 |
| Matting 网络评估 | 用 MODNet/RVM 等 matting 网络重新预测 Alpha，比较与当前 Alpha 的差异 | ⏳ 备选 | 可发现 Alpha 精度不足的区域，但 matting 网络本身也有误差，两个 Alpha 的差异不一定代表溢色；推理开销大（~50ms/帧）；可作为 Lab 色差方案的补充验证 |
| 边缘梯度分析 | 用 Sobel/Canny 检测边缘，分析梯度方向和强度是否符合自然过渡 | ❌ 淘汰 | 梯度分析反映的是亮度/结构变化，非色彩变化；对绿边、色彩溢出无直接检测能力；头发丝等高纹理区域梯度密集导致大量噪声；需要与色彩方法结合才有意义，单独使用不可行 |

**选型依据**：MOS 标注中明显绿边/黑边扣 0.2~0.4，关键判定标准是边缘带是否存在肉眼可见的异常色彩。Lab 色差 ΔE 与人眼色彩感知高度线性相关（CIE 标准），ΔE > 5 可察觉、> 10 明显，这与 MOS 扣分标准可建立直接映射。Alpha 边缘检测精确定位检测区域，避免内部区域干扰。

**输出示例**：
```json
{
  "detector": "EdgeBleed",
  "score": 3.8,
  "confidence": 0.88,
  "evidence": {
    "method": "lab_color_diff_alpha_edge",
    "edge_band_width_px": 5,
    "anomaly_regions": [
      {
        "location": "left_shoulder",
        "bbox": [120, 340, 180, 380],
        "dominant_color_shift": "green",
        "delta_e_mean": 14.2,
        "delta_e_max": 22.7,
        "affected_pixels_ratio": 0.34,
        "interpretation": "左肩区域边缘带 ΔE=14.2，存在明显绿边残留"
      }
    ],
    "overall_edge_quality": {
      "clean_ratio": 0.78,
      "green_bleed_ratio": 0.15,
      "black_bleed_ratio": 0.07
    }
  }
}
```

## 4. HandAnomaly（手部异常检测）

**问题定义**：手部在生成过程中出现结构性异常，表现为多指（6指及以上）、手指粘连/融合、指尖模糊、手指几何比例严重失调（过长/过短/弯折角度异常）。本质是生成模型对手部拓扑结构的建模能力不足，导致关节连接关系和骨骼比例违反人体解剖约束。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| MediaPipe Hands 关键点 + 几何约束校验 | 用 MediaPipe Hands 检测 21 个手部关键点，校验指间角度、骨段长度比、指尖距离等几何约束 | ✅ 选用 | MediaPipe Hands 轻量高效（<10ms/帧），21 点覆盖所有手指关节；几何约束基于人体解剖学先验（指骨比例约 1:0.7:0.5、指间最小距离等），物理意义明确；检测失败本身即为强异常信号（严重畸形 → 关键点无法定位）；可逐指输出异常类型 |
| HRNet 高分辨率关键点检测 | 用 HRNet 等高精度姿态估计模型检测手部关键点 | ❌ 淘汰 | HRNet 精度更高但推理开销大（~40ms/帧），对固定背景场景边际收益不大；HRNet 主要优势在人体姿态（17 点），手部 21 点仍依赖 MediaPipe 或专用手部模型；架构复杂度高，部署维护成本不合理 |
| 骨架化拓扑分析 | 对手部分割 mask 做形态学骨架化，分析骨架分支数（对应手指数）和连通性 | ⏳ 备选 | 对多指检测有独特优势（分支数 > 5 → 多指），不依赖关键点检测器；但对手指粘连（分支合并）和弯曲遮挡（分支消失）场景准确率低；可作为 MediaPipe 关键点方案在多指场景的补充信号 |
| CNN 异常分类器 | 收集手部异常样本（多指、粘连、模糊），训练二分类/多分类器 | ❌ 淘汰 | 同样面临标注数据不足问题；手部异常形态多样（多指位置、粘连方式各异），分类器泛化性差；输出概率值无法告知"第几根手指异常、什么类型异常"，可解释性不足 |

**选型依据**：MOS 标注中手部模糊/粘连扣 0.1~0.3，扣分相对较轻但用户感知明显。MediaPipe Hands 提供的 21 点关键点天然对应人手解剖结构，通过几何约束校验可直接输出"哪根手指、什么异常"的可解释结果。关键点检测置信度本身也是有效信号——严重畸形时检测器直接失败，可作为兜底异常判定。

**输出示例**：
```json
{
  "detector": "HandAnomaly",
  "score": 4.1,
  "confidence": 0.78,
  "evidence": {
    "method": "mediapipe_hands_geometry",
    "hands_detected": 2,
    "anomalies": [
      {
        "hand": "right",
        "type": "finger_adhesion",
        "fingers_involved": ["index", "middle"],
        "tip_distance_px": 3.2,
        "min_normal_distance_px": 12.0,
        "joint_angle_deviation_deg": 28.5,
        "interpretation": "右手食指与中指指尖距离 3.2px，远低于正常最小值 12.0px，判定为手指粘连"
      }
    ],
    "geometry_check": {
      "bone_ratio_valid": true,
      "finger_count": 5,
      "detection_confidence": 0.72
    }
  }
}
```

## 5. TemporalFlicker（时序闪烁检测）

**问题定义**：视频在时间轴上出现不连续性，表现为亮度/色彩闪烁、局部区域抖动、突变帧（某帧突然出现大面积异常后恢复）。本质是帧间生成缺乏时序一致性约束，导致相邻帧在像素层面产生不合理的突变。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| 帧间光流一致性 + 时序方差 | 用 RAFT/Farneback 计算相邻帧光流，统计光流幅值的时序方差；同时计算关键区域像素值的时序方差；两者联合判定闪烁 | ✅ 选用 | 光流捕捉运动一致性，时序方差捕捉亮度/色彩突变，二者互补覆盖"抖动"和"闪烁"两类时序劣化；光流幅值方差直接反映运动平滑度；像素时序方差直接反映亮度稳定性；可逐区域输出闪烁强度热力图 |
| SSIM 帧间差 | 计算相邻帧 SSIM，低 SSIM → 帧间差异大 → 可能闪烁 | ❌ 淘汰 | SSIM 对正常运动（说话、手势）和异常闪烁无法区分——正常运动也会导致 SSIM 下降；需要设定动态阈值来排除合理运动，阈值难以确定；无法区分"全局闪烁"和"局部抖动" |
| 像素级时序方差（纯统计） | 对每个像素位置统计时间窗口内的方差，高方差区域标记为闪烁 | ⏳ 备选 | 实现最简单，可作为光流方案的快速预筛；但无法区分正常运动和异常闪烁（说话嘴部区域方差必然高）；单独使用误报率高；可作为光流方案的辅助通道 |
| 3D CNN 时序分类 | 用 SlowFast/C3D 等 3D CNN 对视频片段做闪烁/正常分类 | ❌ 淘汰 | 需要大量标注的闪烁视频片段（当前无此数据集）；推理开销大（3D 卷积 ~100ms/clip）；黑箱输出无法告知"第几帧、哪个区域闪烁"，可解释性差；对新型闪烁模式泛化能力未知 |

**选型依据**：时序闪烁的核心特征是帧间变化的不合理突增。光流一致性直接度量运动连续性（正常运动光流平滑，抖动光流突变），时序方差直接度量亮度稳定性（正常帧亮度平稳，闪烁帧亮度跳变）。两者联合可覆盖运动类和亮度类时序劣化，且均可输出逐帧、逐区域的量化值。

**输出示例**：
```json
{
  "detector": "TemporalFlicker",
  "score": 3.9,
  "confidence": 0.82,
  "evidence": {
    "method": "optical_flow_consistency_temporal_variance",
    "analysis_window": {"start_frame": 45, "end_frame": 75},
    "flicker_events": [
      {
        "frame_range": [52, 54],
        "type": "brightness_flicker",
        "region": "face",
        "pixel_variance_spike": 1842.5,
        "normal_variance_baseline": 120.3,
        "spike_ratio": 15.3,
        "interpretation": "第52-54帧面部区域像素方差突增15.3倍，存在明显亮度闪烁"
      },
      {
        "frame_range": [61, 62],
        "type": "motion_jitter",
        "region": "left_hand",
        "flow_magnitude_std": 8.7,
        "normal_flow_std": 1.2,
        "interpretation": "第61-62帧左手区域光流幅值标准差异常（8.7 vs 正常1.2），存在运动抖动"
      }
    ],
    "overall_temporal_stability": 0.71
  }
}
```

## 6. BackgroundArtifact（背景异常检测）

**问题定义**：背景区域出现视觉异常，表现为背景模糊/失焦、局部伪影（色块、纹理错误）、前景与背景过渡不自然（硬边界、光照不一致）。本质是背景生成/替换过程中与参考背景的一致性被破坏，或生成质量不足导致非自然纹理。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| 参考帧对比 + SSIM 局部异常 | 以首帧/模板作为背景参考帧，计算每帧背景区域与参考帧的逐块 SSIM；低 SSIM 块标记为异常 | ✅ 选用 | 固定背景场景背景通常固定或缓变，参考帧对比天然合理；SSIM 捕捉亮度、对比度、结构三维度变化，与人眼感知对齐良好；逐块计算可精确定位异常区域；计算轻量（~3ms/帧）；输出 SSIM 值物理含义明确 |
| 背景一致性检测（帧间） | 提取背景区域特征（颜色直方图、纹理描述子），检测帧间一致性突变 | ❌ 淘汰 | 与参考帧对比方案功能重叠但粒度更粗；直方图级统计丢失空间信息，无法定位"哪块区域出问题"；对局部小面积伪影（色块）不敏感 |
| 频域分析（FFT/小波） | 对背景区域做频域分析，检测异常频率分量（如棋盘格伪影对应特定频率峰值） | ⏳ 备选 | 对特定类型伪影（周期性纹理错误、摩尔纹）检测能力强；但对模糊、光照不一致等非频域特征的劣化无感知力；可作为 SSIM 方案在特定伪影类型上的补充 |
| 生成模型判别器 | 用预训练的 GAN 判别器（如 StyleGAN Discriminator）判定背景是否为生成内容 | ❌ 淘汰 | 判别器只区分真/假，不区分好/坏的生成结果；对同一生成模型的输出判别器可能过拟合为"真"；推理开销大；输出单一真假概率无法定位具体异常区域和类型 |

**选型依据**：MOS 标注中背景模糊仅轻微拖累 -0.1~-0.4，属于低优先级但需覆盖的检测项。参考帧对比是固定背景场景的天然优势（背景通常可控），SSIM 逐块分析可精确定位异常区域，输出的 SSIM 值直接反映"该区域与参考背景的一致程度"，与 MOS 扣分幅度可建立线性映射。

## 7. CompressionArtifact（编码压缩劣化检测）

**问题定义**：H.264/HEVC 低码率编码导致 DCT 块边界可见、蚊式噪点、420 色度块化。与生成模型纹理丢失不同，属于 `encoding_loss` 成因。

**候选方案**：

| 方案 | 原理 | 选用/淘汰 | 理由 |
|------|------|-----------|------|
| DCT 块边界能量 + 局部 BRISQUE | 8×8 边界差分能量比；候选区局部 BRISQUE 确认 | ✅ 选用 | 针对 block 伪影；跨 ROI 统一；可输出 blockiness_score |
| 整图 BRISQUE/NIQE | 无参考质量 | ❌ 淘汰 | 合成图像 NSS 先验不成立 |
| 并入 HairTexture only | 仅在头发检 block | ❌ 不采用 | face/edge  block 漏检；报告难合并 |

**选型依据**：badcase 最高频问题之一。独立检测器便于路由与 root_cause 归因。详见 [`COMPRESSION_AND_ENCODING.md`](COMPRESSION_AND_ENCODING.md)。

**输出示例**：
```json
{
  "detector": "compression_artifact",
  "evidence": {
    "method": "dct_blockiness",
    "metric": "blockiness_score",
    "value": 2.1,
    "threshold": 1.8,
    "interpretation": "块边界能量比 2.1 超过阈值，存在 H.264 块效应"
  }
}
```
